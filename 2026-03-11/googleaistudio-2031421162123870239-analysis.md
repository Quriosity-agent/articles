## 技术上这件事为什么重要

过去很多团队做“多模态搜索”时，实际是拼接方案：
- 文本用一个 embedding 模型
- 图片用另一个视觉 encoder
- 视频再单独做帧级特征
- 音频转文本后再走文本 embedding

最后在工程层做映射、融合、重排。这种方案能用，但维护成本很高，跨模态质量也不稳定。

**“原生多模态 embedding + 单一向量空间”** 的意义在于：
1. **统一索引结构**：一个向量库可同时存多种模态特征。
2. **跨模态召回更直接**：例如“文字搜视频片段”“图片搜文档段落”。
3. **分类任务更轻量**：可在 embedding 上做相似度分类、kNN 或轻量头部模型。
4. **系统复杂度下降**：减少多模型编排与跨模型对齐成本。

## Builder 视角：可以马上落地的 3 种场景

### 1) 企业知识库检索（文档+截图+会议录音）
- 把 PDF/Doc、产品截图、培训视频、会议音频统一入库
- 用户可用自然语言提问，系统跨模态召回
- 典型收益：减少“信息在不同媒介里找不到”的问题

### 2) 内容审核与品牌资产管理
- 用同一 embedding 空间做“相似素材”聚类
- 同时检索图像、视频、文案是否偏离品牌规范
- 对增长团队来说，能更快发现“高相似高转化”的素材组合

### 3) 多模态推荐与个性化
- 用用户文本行为 + 图片点击 + 视频观看特征构建统一用户向量
- 在召回层先跨模态粗筛，再用排序模型精排
- 优点是召回覆盖更广，不只依赖单一行为信号

## 实操建议（避免踩坑）

如果你准备试 Gemini Embedding 2，我建议按这个顺序做：

1. **先做小规模离线评测**
   - 建一个 1k~10k 样本的真实数据集
   - 指标看 Recall@K、nDCG、MRR，别只看主观 Demo

2. **明确“跨模态查询”占比**
   - 如果 95% 都是文本查文本，未必需要立刻全量迁移
   - 如果跨模态需求高（图搜文、文搜视频），优先级应拉高

3. **提前设计 chunking 与元数据策略**
   - 视频按镜头/时间窗口切分
   - 文档按语义段落切分
   - 元数据必须有来源、时间、权限标签（后续权限过滤非常关键）

4. **在线架构预留重排层**
   - 统一 embedding 能解决“召回”问题，不等于最终排序最优
   - 最佳实践通常是：向量召回 + cross-encoder 或任务模型重排
