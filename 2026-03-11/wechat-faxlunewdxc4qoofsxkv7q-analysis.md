# 从“会答题”到“能交付”：$OneMillion-Bench 给 AI Agent 创业者的五个硬结论

原文链接：<https://mp.weixin.qq.com/s/FAXLuNewDxc4QoOFsxkV7Q>  
文章来源：红杉汇（内容核心来自 xbench / Humanlaya Data Lab / BIGAI / M-A-P 联合发布）

## 先说结论：AI 已经能“赚钱”，但还没到“放心托付”

这篇文章最有价值的地方，不是又发了一个榜单，而是把评测目标从“分数”切到“可兑现经济价值”。

核心方法很直白：

- 先定义一组真实专家任务（金融、法律、医疗、自然科学、工业）
- 给每个任务标价（资深专家耗时 × 时薪）
- 再看模型是否达到可交付门槛（单题 >=70% 记为通过）
- 通过的任务才计入“能赚到的钱”

在这个口径下：

- 任务池总价值约 **100万美元**
- 顶级模型可交付价值约 **48~50万美元**
- Token 成本约 **100美元量级**

这意味着：AI 在高单价脑力工作中，已经有了惊人的成本杠杆；但如果你要把完整白领流程“全托管”，今天仍然风险很高。

---

## 这个基准为什么重要（而且比一般 benchmark 更“接地气”）

传统 benchmark 常见问题：

- 更像考试，不像工作
- 只看准确率，不看经济回报
- 中文与本地场景覆盖薄弱
- 难以映射到企业 KPI

$OneMillion-Bench 的设计，解决的是“能不能上线”的问题：

1. **高价值任务而非知识问答**  
   400 道题（200 中文 + 200 英文），覆盖 92 个三级领域，强调开放式专家任务。

2. **Rubric 细颗粒评分 + LLM-as-Judge**  
   每题拆成 15–35 个考点，总计 7000+ 考点，不只问“是什么”，还看“怎么做、顺序、理由”。

3. **非对称负分机制（+10 到 -20）**  
   明确打击“说很多看起来很专业”的 reward hacking。现实工作里，错一次的代价本来就可能比做对一次更大。

4. **CN / Global 子集分离**  
   单独建中文本地化任务，能更真实测试法规、流程、语境差异。

5. **专家生产流水线**  
   专家筛选通过率 <5%，题目质检通过率 38.1%，并有对抗评审与仲裁机制，强调一致性与可复核。

这套体系的意义是：让模型评测结果更接近“采购一个 Agent 团队后，财务报表会发生什么”。

## 成绩单怎么读：平均分会误导，Pass Rate 才能指导上线

文章给了一个非常实用的判断框架：

- **平均分** ≈ 考试成绩
- **通过率（>=70%）** ≈ 上岗证

头部模型平均分可能进了“合格区”，但通过率显著掉到 40%+。这代表什么？

- 它可以完成不少片段任务
- 但在真实交付标准下，仍有大量任务需要返工或人工接管

对 builder 来说，这直接对应产品策略：

- 不要把“看起来聪明”当成“可签 SLA”
- 要把评估指标从“平均体验”改成“可交付比例 + 返工成本 + 错误损失”

## 三个技术洞察（对 Agent 产品最有用）

### 1) Web Search 是增益器，也是风险放大器

搜索能显著补齐事实，尤其是时效性知识（金融、法律条文、医疗指南、工业标准）。

但它也会引入：

- 低质量来源污染
- 看似权威但过期/错误的信息
- 推理链不一致

**工程建议：**

- 做来源分级（官方/学术/行业白名单）
- 强制多源交叉验证
- 把“证据链”结构化输出（claim → source → confidence）
- 为高风险领域设置“无法确认即拒答/转人工”

### 2) 真瓶颈不是表达，而是复杂多步推理执行

模型常见问题不是“不会说”，而是：

- 在多步任务中跳步
- 缺少反证与回溯
- 给出方向正确但不可执行的建议

**工程建议：**

- 引入 plan-execute-verify 三阶段 pipeline
- 中间态强约束（每步必须产出可验证 artifact）
- 对关键任务加入“反例检查”和“失败回滚策略”

### 3) 评分机制决定了模型行为

如果你的内部评估只奖励“覆盖点数”，模型会学会“多说”。

**工程建议：**

- 引入非对称惩罚（关键错误高惩罚）
- 按业务后果建损失函数（误诊、误法务建议、错误风控）
- 把“看起来像答案”与“可执行交付”分开评分

---

## 给 Builder 的落地清单（可直接抄到 roadmap）

### A. 评估侧

- 从“准确率”切到“可交付率（Pass Rate）”
- 按任务经济价值做加权，不要一题一票
- 建立高风险负分项，避免 KPI 被“话术能力”骗分

### B. 产品侧

- 先做“半自动可交付”，不要直接承诺“全自动替代”
- 把工作流拆成：可自动 / 需审核 / 必须人工 三层
- 在 UI 里暴露证据链与不确定性，而不是只给一个漂亮答案

### C. 商业侧

- 用“节省专家工时”而非“模型分数”讲 ROI
- 合同承诺基于可验收任务比例，而非单次 demo 效果
- 把返工率、人工兜底成本、事故率纳入单位经济模型

---

## 我的判断

$OneMillion-Bench 的价值不在于告诉你“谁第一”，而在于把 Agent 讨论从“智能幻觉”拉回“交付现实”。

如果你在做垂直 Agent，这个 benchmark 给了一个很明确的方向：

- 不是追求更花哨的回答，
- 而是追求更高比例、可审计、可复核、可控风险的真实交付。

当你能稳定把“通过率 × 任务单价”做上去，AI 才真正从工具变成生产力资产。
