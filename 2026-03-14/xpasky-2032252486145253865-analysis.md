# Chrome 146 正式支持 MCP：一个开关，让 AI Agent 接管你的浏览器

> 原推：[@xpasky](https://x.com/xpasky/status/2032252486145253865) | 551 likes, 36 RT | 2026-03-13

## 发生了什么

Chrome 146 昨天正式发布，内置了 MCP（Model Context Protocol）支持。这意味着：

**你只需要打开一个开关，就能把当前浏览器会话通过 MCP 暴露给 CLI agent。**

Agent 可以直接操作你已经登录的网站——不需要单独配置认证、不需要 cookie 导出、不需要 headless 浏览器。你在浏览器里登录了 Gmail，agent 就能帮你处理邮件。

![Chrome MCP 设置界面](https://pbs.twimg.com/media/HDQB-gWWYAAHyDN.jpg)
*图片来源：[@xpasky](https://x.com/xpasky/status/2032252486145253865)*

## 为什么这很重要

之前让 AI agent 操作浏览器，你得：
1. 配置 headless Chrome + CDP
2. 处理登录态（导出 cookie、配置 OAuth）
3. 维护一套独立的浏览器自动化环境

现在？**一个 toggle，完事。**

作者 Petr Baudis 早在今年 1 月就预言了这个功能（当时 Chrome 144 还在 early stable），原话是：

> "I should be able to just `amp mcp add chrome-devtools -- npx chrome-devtools-mcp@latest --autoConnect` and let Claude browse on my behalf, within my login sessions."

两个月后，Chrome 146 把这个变成了现实。

## 实际用法

从截图来看，Chrome 的设置里新增了 MCP 相关选项，开启后会暴露一个 MCP endpoint。CLI agent（比如 Claude Code、Codex 等）可以通过标准 MCP 协议连接，直接操作浏览器：

- 在你已登录的 LinkedIn 里处理连接请求（作者就是这么用的）
- 在 Gmail 里搜索和回复邮件
- 在任何 SaaS 工具里执行操作

## Builder 视角

这对 agent 开发者意味着什么：

1. **认证问题直接消失** — 不再需要为每个网站单独处理登录流程，用户的浏览器会话就是最好的认证
2. **MCP 成为浏览器自动化的标准接口** — Chrome 官方支持意味着这不是实验性功能，是正式的基础设施
3. **Agent 的能力边界大幅扩展** — 任何你能在浏览器里做的事，agent 都能做

## 注意事项

- 这是 Chrome 146+，确认你的浏览器已更新
- 安全性需要考虑：你在暴露整个浏览器会话，确保只在信任的 agent 上使用
- 目前主要面向开发者/power user，普通用户可能还需要等更好的 UX

## 一句话总结

Chrome 原生 MCP 支持 = AI agent 获得了"真人级别"的浏览器操作能力。不是模拟，是直接用你的登录态。这是 agent 基础设施的一个里程碑。
