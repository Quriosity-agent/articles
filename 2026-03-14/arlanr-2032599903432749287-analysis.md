## 一句话总结

Nia CLI 是一个终端工具，让 AI Agent 能索引、搜索、深度研究各种数据源（论文、代码库、文档、Slack、Google Drive 等），还能保存对话历史跨 Agent 复用。

## 为什么值得关注

现在做 Agent 开发最头疼的问题之一：**Agent 没有好的知识检索能力**。你可以给 Agent 一堆工具，但如果它不能高效地搜索你的代码库、内部文档、API 文档，那它的能力上限就很低。

Nia CLI 解决的就是这个问题。

## 四大核心功能

### 1. Index（索引）
把各种数据源建立索引：
- 论文、技术文档
- 代码库
- 内部 Wiki
- 还有更多

### 2. Search（搜索）
在已索引的数据源上做 Agent 级别的搜索：
- 文档和 API
- Slack 消息
- Google Drive 文件

### 3. Research（深度研究）
跨整个 GitHub 仓库或个人数据做深度研究。这不是简单的关键词搜索，而是理解上下文的智能研究。

### 4. Save（保存）
保存对话历史和计划，在另一个 Agent 中恢复。这意味着：
- Agent A 的研究成果可以无缝传给 Agent B
- 不会丢失上下文线索

## 快速开始

一行命令搞定：

```bash
bunx nia-wizard@latest
```

用 `bunx` 直接运行，不用全局安装，非常干净。

## 实际应用场景

1. **代码库理解**：新加入一个大项目，让 Agent 先索引整个代码库，然后随时搜索任何模块的实现细节
2. **文档查询**：把团队的 Confluence/Notion 索引起来，Agent 可以直接回答"我们的部署流程是什么"
3. **跨平台研究**：同时搜索 Slack 对话和 Google Drive 文档，找到分散在各处的信息
4. **Agent 协作**：一个 Agent 做完前期研究，保存状态，让另一个专门的 Agent 继续执行

## 我的看法

Agent 工具生态正在从"能用"走向"好用"。Nia CLI 瞄准的是一个真正的痛点——Agent 的知识检索层。当 Agent 能高效地搜索和理解你的数据源时，它的输出质量会有质的提升。

值得试试，特别是如果你在构建需要大量上下文的 Agent 工作流。

## 更新：GitHub 仓库源码分析

Nia CLI 的 GitHub 仓库 [nozomio-labs/nia-cli](https://github.com/nozomio-labs/nia-cli) 现已公开，Apache 2.0 许可证。以下是源码层面的深度分析。

### 仓库概览

- **包名：** `@nozomioai/nia` v0.0.7
- **运行时：** Bun (>=1.3.10)，不是 Node.js
- **语言：** TypeScript（ESNext，strict 模式）
- **CLI 框架：** Crust（`@crustjs/core`）— Nozomio 自研的 CLI 框架
- **API 客户端：** `nia-ai-ts` — TypeScript SDK，生成的 OpenAPI 客户端
- **唯一的运行时依赖：** `chokidar` v5（文件监听）

### 项目结构

```

│   ├── auth.ts         # 认证管理
│   ├── search.ts       # 搜索（4 个子命令）
│   ├── repos.ts        # 仓库索引管理（8 个子命令）
│   ├── sources.ts      # 文档源管理
│   ├── local.ts        # 本地文件夹同步
│   ├── oracle.ts       # 自主研究（Oracle）

│   ├── datasets.ts     # 数据集索引

├── services/
```

### 架构设计亮点

**1. 瘦客户端 + 云端重活**

CLI 本身是个非常薄的客户端。所有索引、搜索、AI 处理都在 Nia 的云端 API 完成（`https://apigcp.trynia.ai/v2`）。CLI 只负责：
- 参数解析和验证
- API 调用编排
- 输出格式化
- 本地文件夹的监听和同步上传

这意味着 CLI 安装体积很小，更新迭代快。

**2. Crust CLI 框架**

Nia 没有用 Commander.js 或 yargs 这些常见方案，而是用了自研的 Crust 框架（`@crustjs/core`）。这个框架支持：
- 链式子命令构建（`app.sub("name").args([...]).flags({...}).run()`）
- 全局 flags 自动继承（`inherit: true`）
- 插件系统（版本检查、自动补全、帮助文档）
- **最关键的：`@crustjs/skills` 插件** — 给 AI Agent 生成结构化的工具描述

```typescript
// 用 annotate() 给每个命令添加 Agent 使用说明
const queryCommand = annotate(
  app.sub("query")
    .meta({ description: "Query indexed repositories" })
    .run(async ({ args, flags }) => { ... }),
  [
    "Targeted search with AI response and sources.",
    "Use --skip-llm to return raw results without AI synthesis.",
  ],
);
```

这个 `annotate()` 是整个设计中最聪明的地方——每个 CLI 命令都自带「Agent 应该怎么用我」的说明。

**3. 内嵌的 Agent 工作流指令**

`cli.ts` 里的 `skillPlugin` 配置了一套完整的 Agent 使用规范：

> "Use indexed Nia sources first. Web is the fallback, not the default."

规定了 Agent 必须遵循的搜索优先级：
1. 先检查已索引的数据源（`nia repos list`）
2. 如果已索引，在 Nia 内搜索（`nia search query`）
3. 如果未索引但知道标识符，先索引再搜索
4. 只有以上都不行，才用 web 搜索

这确保了 Agent 优先使用已索引的高质量数据，而不是每次都跑去 web 搜索。

### 搜索系统实现

搜索分四层，从轻到重：

| 命令 | 用途 | 特点 |
|------|------|------|
| `search universal` | 全局语义搜索 | 混合向量 + BM25，不含 Slack |
| `search query` | 精准搜索 | 支持指定 repos/docs/本地文件夹 |
| `search web` | Web 搜索 | 7 种分类过滤（github/news/tweet 等） |
| `search deep` | 深度研究 | 多步推理，需要 Pro 计划 |

`search query` 的搜索模式会自动推断：
```typescript
// 只传了 --repos → repositories 模式
// 只传了 --docs 或 --local-folders → sources 模式
// 混合传入 → unified 模式
```

还支持 `--fast`（跳过 LLM 处理，100-500ms 返回）、`--skip-llm`（返回原始结果）、`--strategy`（vector/tree/hybrid 三种检索策略）。

### 仓库索引：比你想的丰富

`repos` 命令不只是 `index` + `list`，还包括：
- `repos tree` — 查看仓库文件树（支持路径/扩展名过滤）
- `repos read` — 直接读取索引仓库中的文件内容（带行号）
- `repos grep` — 在仓库中做 regex 搜索（支持上下文行数、exhaustive 模式）
- `repos rename` / `repos delete` — 管理已索引的仓库

这意味着 Agent 可以像操作本地文件一样操作远程仓库，而不需要 clone。

### 跨 Agent 上下文保存（Save/Resume）

`contexts` 是之前文章提到的 Save 功能的具体实现。源码揭示了更多细节：

**四种记忆类型：**
- `scratchpad` — 临时草稿
- `episodic` — 事件记录
- `fact` — 事实知识
- `procedural` — 操作流程

**支持 TTL：** 上下文可以设置过期时间（秒），自动清理。

**支持 stdin 管道：** `--content '-'` 可以从 stdin 读取内容，方便和其他命令链式使用。

**三种搜索方式：**
- `contexts search` — 文本搜索（标题、摘要、内容、标签）
- `contexts semantic` — 语义搜索（向量 + BM25 混合）
- `contexts list` — 带过滤的分页列表

### 本地文件夹同步

`local` 命令让 Nia 能实时监听本地文件变化并同步到云端索引：

```
nia local add ~/dev/my-project   # 注册文件夹
nia local sync                    # 一次性同步
nia local watch                   # 持续监听
```

`watcher.ts` 用 chokidar v5 实现文件监听，带 2 秒防抖和写入完成检测：
```typescript
chokidar.watch(folderPath, {
  ignoreInitial: true,
  awaitWriteFinish: {
    stabilityThreshold: 500,
    pollInterval: 100,
  },
});
```

`watch` 模式还有个巧妙的设计：每 30 秒刷新源列表（自动发现新注册的文件夹），每 600 秒做一次 fallback 全量同步（防止文件事件漏掉）。

### Oracle（自主研究）

Oracle 是 Nia 最重的功能——异步执行的多步研究任务。源码显示：
- 创建研究任务后返回 `job_id`，不会阻塞
- 支持 SSE 流式输出研究进度（`oracle stream`）
- 支持交互式会话（`oracle chat`）和非交互式管道（`--message` + `--no-interaction`）
- 可以指定搜索范围（`--repos`、`--docs`）和输出格式
- 支持 cancel、status 查询

### 值得注意的技术选择

1. **Bun 而非 Node.js：** 用 `bun run build` 编译成单个可执行文件，性能和启动速度都更好
2. **Biome 而非 ESLint：** 更快的 lint 和格式化
3. **Changesets 管理版本：** 用 `@changesets/cli` 管理语义化版本
4. **API key 三级解析链：** CLI flag > 环境变量 > 配置文件，灵活且安全
5. **所有 API 调用通过 OpenAPI 生成的类型安全客户端：** 不手写 fetch，减少出错

### 总结

看完源码，Nia CLI 的定位更清晰了：**它不是一个搜索引擎，而是一个给 AI Agent 用的知识检索 SDK**。整个设计——从 `annotate()` 给命令添加 Agent 说明，到 `skillPlugin` 定义 Agent 工作流，到上下文的四种记忆类型——都是为 Agent 优先设计的。

对开发者来说，CLI 界面很友好。但 Nia 真正的价值在于：它让任何 Agent 框架都能用一行 `bunx nia-wizard@latest` 获得完整的知识检索能力。
