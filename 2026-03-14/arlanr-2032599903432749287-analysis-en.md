## TL;DR

Nia CLI is a terminal tool that lets AI agents index, search, and deep-research across data sources — papers, codebases, docs, Slack, Google Drive — and save conversation state across agents.

## Why This Matters

One of the biggest bottlenecks in agent development: **agents can't efficiently search your actual data**. You can give an agent dozens of tools, but if it can't search your codebase, internal docs, or API references effectively, its ceiling is low.

Nia CLI tackles this head-on.

## Four Core Capabilities

### 1. Index
Build indexes over diverse data sources:
- Papers and technical docs
- Codebases
- Internal wikis

### 2. Search
Agentic search across indexed sources:
- Documentation and APIs
- Slack messages
- Google Drive files

### 3. Research
Run deep research across entire GitHub repositories or personal data. Not keyword matching — context-aware intelligent research.

### 4. Save
Save conversation histories and plans. Resume them in another agent without losing the thread. This enables:
- Seamless handoff between specialized agents
- No context loss between sessions

## Getting Started

One command:

```bash
bunx nia-wizard@latest
```

Runs via `bunx` — no global install needed.

## Practical Use Cases

1. **Codebase onboarding**: Index an entire repo, then query any module's implementation details on demand
2. **Doc retrieval**: Index your team's Confluence/Notion, let agents answer "what's our deploy process?"
3. **Cross-platform research**: Search Slack conversations and Google Drive simultaneously
4. **Agent handoff**: One agent does research, saves state, another picks up execution

## Take

The agent tooling ecosystem is maturing from "works" to "works well." Nia CLI targets a real pain point — the knowledge retrieval layer for agents. When agents can efficiently search and understand your data, output quality improves dramatically.

Worth trying if you're building context-heavy agent workflows.

## Update: GitHub Repository Source Code Analysis

The Nia CLI GitHub repo [nozomio-labs/nia-cli](https://github.com/nozomio-labs/nia-cli) is now public under Apache 2.0. Here's what the source code reveals.

### Repository Overview

- **Package:** `@nozomioai/nia` v0.0.7
- **Runtime:** Bun (>=1.3.10) — not Node.js
- **Language:** TypeScript (ESNext, strict mode)
- **CLI Framework:** Crust (`@crustjs/core`) — Nozomio's own CLI framework
- **API Client:** `nia-ai-ts` — generated OpenAPI TypeScript SDK
- **Only runtime dependency:** `chokidar` v5 (file watching)

### Architectural Highlights

**1. Thin Client, Heavy Cloud**

The CLI itself is an extremely thin client. All indexing, search, and AI processing happens on Nia's cloud API (`https://apigcp.trynia.ai/v2`). The CLI handles:
- Argument parsing and validation
- API call orchestration
- Output formatting
- Local folder watching and sync uploads

This keeps the install size small and iteration cycles fast.

**2. Crust CLI Framework**

Instead of Commander.js or yargs, Nia uses their own Crust framework (`@crustjs/core`). It supports:
- Chainable subcommand building (`app.sub("name").args([...]).flags({...}).run()`)
- Global flag inheritance (`inherit: true`)
- Plugin system (update notifier, autocomplete, help generation)
- **The key feature: `@crustjs/skills` plugin** — generates structured tool descriptions for AI agents

```typescript
// annotate() adds agent-facing usage hints to each command
const queryCommand = annotate(
  app.sub("query")
    .meta({ description: "Query indexed repositories" })
    .run(async ({ args, flags }) => { ... }),
  [
    "Targeted search with AI response and sources.",
    "Use --skip-llm to return raw results without AI synthesis.",
  ],
);
```

This `annotate()` pattern is the smartest design decision — every CLI command carries its own "how an agent should use me" documentation.

**3. Embedded Agent Workflow Instructions**

The `skillPlugin` in `cli.ts` defines a complete agent usage protocol:

> "Use indexed Nia sources first. Web is the fallback, not the default."

It mandates a strict search priority for agents:
1. Check existing indexed sources (`nia repos list`)
2. If indexed, search within Nia (`nia search query`)
3. If not indexed but identifier is known, index first then search
4. Only fall back to web search as a last resort

This ensures agents prioritize high-quality indexed data over ad-hoc web searches.

### Search System Implementation

Four tiers, from light to heavy:

| Command | Purpose | Characteristics |
|---------|---------|-----------------|
| `search universal` | Global semantic search | Hybrid vector + BM25, excludes Slack |
| `search query` | Targeted search | Supports specific repos/docs/local folders |
| `search web` | Web search | 7 category filters (github/news/tweet etc.) |
| `search deep` | Deep research | Multi-step reasoning, Pro plan required |

`search query` auto-detects the search mode:
```typescript
// Only --repos passed → repositories mode
// Only --docs or --local-folders → sources mode
// Mixed → unified mode
```

Also supports `--fast` (skip LLM, 100-500ms response), `--skip-llm` (raw results), and `--strategy` (vector/tree/hybrid retrieval strategies).

### Repository Indexing: More Than You'd Expect

The `repos` command goes well beyond `index` + `list`:
- `repos tree` — view repository file tree (with path/extension filters)
- `repos read` — read file content from indexed repos (with line numbers)
- `repos grep` — regex search across repo files (context lines, exhaustive mode)
- `repos rename` / `repos delete` — manage indexed repositories

This means an agent can operate on remote repos like local files — no clone required.

### Cross-Agent Context Save/Resume

`contexts` is the concrete implementation of the Save feature from the original article. The source code reveals more detail:

**Four memory types:**
- `scratchpad` — temporary working notes
- `episodic` — event records
- `fact` — factual knowledge
- `procedural` — operational procedures

**TTL support:** Contexts can have expiration times (in seconds) for automatic cleanup.

**Stdin piping:** `--content '-'` reads from stdin, enabling command chaining.

**Three search modes:**
- `contexts search` — text search (title, summary, content, tags)
- `contexts semantic` — semantic search (hybrid vector + BM25)
- `contexts list` — paginated listing with filters

### Local Folder Sync

The `local` command lets Nia watch local file changes and sync to the cloud index in real-time:

```
nia local add ~/dev/my-project   # Register a folder
nia local sync                    # One-time sync
nia local watch                   # Continuous watch
```

`watcher.ts` uses chokidar v5 with 2-second debounce and write-finish detection:
```typescript
chokidar.watch(folderPath, {
  ignoreInitial: true,
  awaitWriteFinish: {
    stabilityThreshold: 500,
    pollInterval: 100,
  },
});
```

The `watch` mode has a clever design: it refreshes the source list every 30 seconds (auto-discovers newly registered folders) and does a fallback full sync every 600 seconds (catches missed file events).

### Oracle (Autonomous Research)

Oracle is Nia's heaviest feature — asynchronous multi-step research tasks. Source code shows:
- Creating a research task returns a `job_id` without blocking
- SSE streaming for real-time progress (`oracle stream`)
- Interactive sessions (`oracle chat`) and non-interactive piping (`--message` + `--no-interaction`)
- Scoped research (`--repos`, `--docs`) and output format hints
- Cancel and status queries supported

### Notable Technical Choices

1. **Bun over Node.js:** `bun run build` compiles to a single executable — better perf and startup time
2. **Biome over ESLint:** Faster linting and formatting
3. **Changesets for versioning:** `@changesets/cli` for semantic version management
4. **Three-tier API key resolution:** CLI flag > env var > config file — flexible and secure
5. **All API calls via generated type-safe OpenAPI client:** No hand-written fetch calls, fewer bugs

### Takeaway

After reading the source, Nia CLI's positioning becomes clearer: **it's not a search engine — it's a knowledge retrieval SDK designed for AI agents**. The entire design — from `annotate()` adding agent hints to commands, to `skillPlugin` defining agent workflows, to the four memory types for context sharing — is agent-first.

For developers, the CLI interface is clean and friendly. But Nia's real value is this: it lets any agent framework gain full knowledge retrieval capabilities with a single `bunx nia-wizard@latest`.
