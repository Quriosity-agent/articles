## 线程的技术意义

### 1）父推风格：描述力强，但冗余高
父推（underwoodxie96）把视觉约束、情绪叙事、氛围语言、negative prompt 全塞在一起。优点是“想象力完整”，缺点是：
- 语义重复（同一约束换不同措辞反复出现）；
- 含糊词较多（模型解释不稳定）；
- token 成本高，迁移和复用困难。

### 2）Emily 风格：更像生产模板
Emily 的版本明显更偏执行层：
- 字段短且边界清晰；
- 对象层级稳定（可程序化检查）；
- 排除项明确；
- 更容易 A/B 测试。

这会直接提升：
- 可复现性
- 调试效率
- 多模型兼容性
- 协作可交接性

## 给 Builder 的可落地建议

### A. 先定 schema，再写内容
先固定结构，再填值。建议至少包含：
- 构图（composition）
- 主体身份（identity）
- 服装与姿态（wardrobe/pose）
- 光线（lighting）
- 镜头（camera）
- 场景（environment）
- 输出风格（output）
- 排除项（exclude）

原则：**一字段一意图**，不要在同一字段里塞叙事和技术参数。

### B. 给提示词做“去 slop” lint
生成前做 30 秒清洗：
1. 删重复形容词；
2. 删不能映射到像素的空泛句；
3. 情绪词控制在 1-2 个；
4. 把 prose 改成可测参数（焦段、色温、角度、比例）。

### C. 创意简报与执行提示词分层
分成两份文档：
- **Creative Brief（给人看）**
- **Execution Prompt（给模型跑）**

不要把“故事文案 + 执行参数”长期混在一个巨型 prompt 里。

### D. 把 prompt 当版本化资产
把 prompt 放进仓库，配合生成结果做 diff 和注释。长期来看，你会得到可复用的“高命中模板库”，而不是一次性玄学。

## 结论
Emily 这条推文本质上在强调一件事：

> **提示词工程不是“写得更文艺”，而是“写得更可控”。**

对于做图像工作流的 Builder，这意味着从“偶尔好看”走向“稳定可交付”。
