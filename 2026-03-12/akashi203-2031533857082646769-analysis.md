# 一夜 320 次实验：AutoKernel 把 GPU Kernel 优化做成了「自动科研循环」

> 来源推文：<https://x.com/Akashi203/status/2031533857082646769?s=20>  
> 相关仓库：<https://github.com/RightNow-AI/autokernel>

![AutoKernel 推文配图](assets/akashi203-2031533857082646769/tweet-image.jpg)
*图：Akashi203 推文原生图片（来源：X / @Akashi203）*

这条推文最值得 Builder 关注的，不是“涨了多少倍”，而是它把 GPU kernel 优化流程产品化了：

- 你给一个 PyTorch 模型；
- 系统自动 profile 出瓶颈 kernel；
- 自动写 Triton/CUDA 替代；
- 固定 benchmark + correctness 检查；
- 保留或回滚，继续下一轮。

一句话：**把“凭经验调 kernel”变成“可持续跑的自动实验系统”。**

---

## 1) 推文核心信息（先看硬指标）

作者 Jaber（@Akashi203）在推文里给出的关键信息：

- 采用和 Karpathy autoresearch 类似的循环思路
- 95 次实验，指标从 **18 TFLOPS 到 187 TFLOPS**
- 声称对比 cuBLAS 达到 **1.31x**
- 覆盖 9 类 kernel（matmul、flash attention、fused mlp、layernorm、rmsnorm、softmax、rope、cross entropy、reduce）
- 每小时约 40 次实验，过夜约 320 次

这不是单点“神优化”，而是一个可复用的优化生产线。

---

## 2) 线程上下文：社区反馈暴露了一个很真实的问题

在同一条对话里，有开发者质疑 benchmark 口径，认为可能存在测量误差或 CPU 限制；作者随后回复：

> “yes i think it hallucinated, i will run more benchmarks tonight and i will share the results tomorrow”

这段很关键，说明两件事：

1. **自动优化系统也会“跑偏”**，尤其在性能评估阶段；
2. **作者愿意公开复核**，这是工程可信度的一部分。

对 Builder 来说，这反而是加分项：愿意把问题暴露出来，才有机会形成稳定评测基线。

---

## 3) 从仓库看架构：它不是“一个脚本”，而是完整闭环

从 `autokernel` 仓库公开信息看，它把流程拆成了几个独立步骤：

- `profile.py`：找模型里最耗时 kernel
- `extract.py`：把 bottleneck 提取成可单独优化的 kernel
- `bench.py`：固定基准 + 5 阶段正确性检查
- `orchestrate.py`：用 Amdahl’s Law 决定先优化谁
- `verify.py`：回插模型做端到端验证

这套分层的价值是：**每一步都能单测、复跑、定位错误**，不是黑箱“让 AI 自己试试”。

---

## 4) 为什么这个方向值得重视

我认为 AutoKernel 的真正意义有三点：

### A. 把优化从“手艺活”变成“搜索问题”

很多 kernel 优化本质是巨大搜索空间（tile size、并行映射、memory access pattern、fusion 策略）。
传统做法靠专家经验；这个项目把它变成自动化迭代搜索。

### B. 把“局部提速”放进系统级收益函数

用 Amdahl’s Law 做调度，意味着目标不再是某个 kernel 跑分漂亮，而是**整体模型吞吐提升最大化**。

### C. 把 Agent 从“写代码”推进到“跑实验 + 做决策”

它强调 keep/revert 和固定 benchmark，这才是科研/工程自动化真正该有的“守门机制”。

---

## 5) 给 Builder 的落地建议（可直接抄）

如果你想在自己项目复用这套方法，可以先做一个最小版：

1. **先固定评测协议**（输入 shape、warmup、重复次数、统计口径）
2. **先做 correctness gate，再追性能**
3. **每次只改一个变量**（否则无法归因）
4. **保留 keep/revert 机制**（失败必须可回滚）
5. **按系统瓶颈排序任务**（不是按“看起来好优化”排序）

最容易踩的坑就是：

- benchmark 噪声太大，误把随机波动当优化；
- 指标定义不稳定，导致版本间不可比；
- 只看单 kernel 峰值，不看端到端收益。

---

## 6) 一个理性结论

这条推文最有价值的不是“某个数字是否最终成立”，而是它演示了一个方向：

**GPU kernel 优化正在从“专家手工调参”进入“自动化实验工厂”阶段。**

如果你正在做模型推理加速、训练系统、或者 Agentic compiler/toolchain，这个项目值得持续跟踪，尤其是后续复核 benchmark 和更多硬件上的复现结果。
